## ⚠️ Issues found

### WR-07 (confirmed in production) — `/submit_result` does not cross-validate winner vs. score
**Severity:** Medium–High (data integrity).
A direct POST of `result=challenger_wins` with set scores `1:6 2:6` (i.e. the challenger clearly *lost*) was
**accepted**. The stored record is self-contradictory:

```
challenger: Testspieler Eins | opponent: F. Binder | result: challenger_wins | score: "1:6 2:6"
```

…and the "winner" was promoted (rank 42 → 13). The thorough client-side validation in `admin.html`
(`validateMatchResult`) prevents this in the browser, but it is **bypassable** by any direct HTTP call.
**Recommendation:** validate `result` against an allowlist *and* recompute the winner from the set scores
server-side in `submit_result`, rejecting on mismatch — before any DB write. (Matches the existing REVIEW.md WR-07.)

### IN-04 (confirmed) — default password disclosed
New-player creation returns `new_player_password: "DefaultPassword1!"` (a fixed literal) plus the generated
username in the JSON response, and there is no forced change on first login. Low severity / by-design for the
admin modal, but a per-player random token would be safer. (Matches REVIEW.md IN-04.)

### Minor — `/submit_result` returns HTML, not JSON
Success and error responses are HTML redirects to `/admin` rather than structured JSON, so programmatic callers
cannot reliably distinguish success from validation failure (related to WR-07).

## ⚠️ Operational issue found & fixed during this round (not an app bug)

The Round-1 DB restore copied `tennis.db` into the volume via a root one-off container, leaving the file
**owned by `root`**. The app runs as `appuser` (uid 1000), so it could read but **not write** — every write
(`/challenge`, `/submit_result`, toggles) failed with `sqlite3.OperationalError: attempt to write a readonly
database`. This had been latent on production since the Round-1 restore. Because the containers are hardened
(`cap_drop: ALL`), even `root` inside the container cannot `chown`; it was fixed from a one-off container
(`chown 1000:1000 /data/tennis.db`, restart). **Lesson:** any DB file placed into the volume out-of-band must be
`chown`ed to uid 1000. The Round-2 restore applied the correct ownership, and a post-restore write test confirmed
writes succeed.

# Round 3 — DB import hardening on v3.63 (WR-04)

**Time:** 2026-06-18 · **Build:** v3.63 (`APP_VERSION = "3.63"`, footer `Version: 3.63 • 18. Juni 2026`)
**Change under test:** `import_database` now uses the SQLite **backup API** (`src.backup(dst)`) instead of
`shutil.copy2`, and validation additionally requires the `app_settings` table and `completed_challenges_view`
(commit shipped in v3.63). **Procedure:** pushed `docker`, built multi-arch, deployed (`compose pull && up -d`,
healthy, `APP_VERSION="3.63"`, DB owned by `appuser`). Fresh backup (`~/tennis.preimport-v363.db`, 41/119) → test.

## Why this round happened
Round-1/2 testing showed the import endpoint *worked* but could not be called "always-safe":
- **WR-04** — it replaced the live DB with `shutil.copy2`, which overwrites the file under any concurrent
  reader connection and can corrupt it mid-write.
- The schema check only validated `players`/`challenges`, so a structurally-incompatible `.db` could import
  "successfully" and break at runtime.

## Results

| Test | Result |
|---|---|
| Export current DB (`/api/settings/db/export`) | 200, 86 KB ✓ |
| **Import under concurrent load** — import fired with **25 parallel `/api/realtime_data` reads** | success ✓ |
| **Concurrent reads during import** | all returned **41 players, zero errors** (`[41]`) — no torn/malformed reads ✓✓ |
| Data intact after import | 41 players, 36 matches, Harbarth #1 ✓ |
| Write after import (toggle availability) | 200 — imported DB writable ✓ |
| Reject garbage `.db` (integrity fail) | 400 ✓ |
| Reject wrong extension (`.txt`) | 400 ✓ |
| **Stricter schema check** (unit test: full vs partial DB) | full → ACCEPTED; partial without `app_settings`/view → **REJECTED** ✓ |

The concurrency row is the WR-04 fix proving out: under the old `shutil.copy2`, a read during the copy could
have hit *"database disk image is malformed"*; with the backup API all 25 concurrent reads stayed consistent.

## Round 3 conclusion
The DB import is now **always-safe** for its intended use: replacing the live DB via the page-by-page,
lock-aware SQLite backup API (safe under concurrent traffic), with up-front rejection of invalid files and
structurally-incompatible schemas. Importing a `.db` exported from this app — even while the site is in use —
no longer risks corrupting other requests. Production data backed up and verified unchanged (round-trip imported
identical data; 41 players / 36 matches confirmed).
