Was du vorbereitest

Überblick, damit nichts in falscher Reihenfolge passiert.

Du baust folgende Kette: Internet → FritzBox (Portfreigabe) → Raspberry Pi → Caddy-Container (HTTPS) → App-Container (Flask). DuckDNS sorgt dafür, dass dein Name (z. B. nechvatal.duckdns.org) immer auf deine wechselnde Heim-IP zeigt, und Caddy holt automatisch ein gültiges Let's-Encrypt-Zertifikat.

Hardware-Checkliste

  • Raspberry Pi 5 (4 GB oder 8 GB)
  • Offizielles 27 W USB-C Netzteil (Pi 5 braucht es wirklich)
  • microSD-Karte ≥ 32 GB (oder besser: NVMe-SSD via HAT)
  • Netzwerkkabel zur FritzBox (LAN ist stabiler als WLAN für einen Server)
  • Ein zweiter PC mit dem Raspberry Pi Imager

Was du dir vorab besorgst

  • Einen kostenlosen DuckDNS-Account (Login per Google/GitHub) → liefert Subdomain + Token
  • Zugang zur FritzBox-Oberfläche (http://fritz.box) als Admin
  • Eine E-Mail-Adresse für die Let's-Encrypt-Registrierung
💡
Die ganze App läuft in zwei Containern. Du musst auf dem Pi kein Python, kein gunicorn und kein Caddy nativ installieren — Docker bringt alles mit. Die alte systemd-Variante aus früheren Anleitungen wird nicht mehr benötigt.
1

Raspberry Pi OS installieren

Mit dem Raspberry Pi Imager auf die SD-Karte / SSD schreiben.

  1. Lade den Raspberry Pi Imager auf deinem PC herunter.
  2. Wähle: Device = Raspberry Pi 5, OS = Raspberry Pi OS (64-bit) (Lite reicht — kein Desktop nötig), Storage = deine Karte/SSD.
  3. Klicke auf das Zahnrad / „Einstellungen bearbeiten" und setze vorab:
Hostname: stechertennis
Benutzername: stecher
Passwort: ein starkes Passwort
SSH: aktivieren (Passwort-Authentifizierung)
WLAN/Locale: optional, Zeitzone Europe/Vienna

Schreibe das Image, stecke die Karte in den Pi, verbinde LAN-Kabel und Strom. Nach ~1 Minute ist er im Netz.

Per SSH einloggen

auf deinem PC
# Hostname funktioniert meist direkt; sonst die IP aus der FritzBox nehmen
ssh stecher@stechertennis.local
✅
Findest du den Pi nicht? Schau in der FritzBox unter Heimnetz → Netzwerk nach dem Gerät stechertennis und nutze dessen IP.
1b

SSH absichern (Schlüssel + Härtung)

Pflicht, sobald der Server öffentlich erreichbar ist. Reihenfolge genau einhalten.

1) Schlüsselpaar auf deinem PC erzeugen

auf deinem PC
ssh-keygen -t ed25519 -C "stecher-tennis"   # ed25519 = modern & sicher
ssh-copy-id stecher@stechertennis.local     # public key auf den Pi kopieren
🚪
Jetzt testen, bevor du Passwörter abschaltest: In einem zweiten Terminal ssh stecher@stechertennis.local — der Login muss ohne Passwort klappen. Erst danach weiter. Sonst sperrst du dich aus.

2) SSH-Server härten

auf dem Pi
sudo nano /etc/ssh/sshd_config.d/99-hardening.conf
99-hardening.conf
Port 10115                       # abweichender Port (optional, gegen Scan-Lärm)
AddressFamily inet               # nur IPv4
PermitRootLogin no
PubkeyAuthentication yes
PasswordAuthentication no        # Passwort-Login AUS
KbdInteractiveAuthentication no
AuthenticationMethods publickey
MaxAuthTries 3
MaxSessions 4
LoginGraceTime 30
prüfen & neu laden
sudo sshd -t                     # Syntax prüfen — muss fehlerfrei sein
sudo systemctl restart ssh
# neuer Login: ssh -p 10115 stecher@stechertennis.local
🧱
Wenn du Port 10115 nutzt: leg dir auf dem PC einen Alias in ~/.ssh/config an (Host pi / Port 10115). Leg keine SSH-Portfreigabe in der FritzBox an — SSH bleibt LAN-only; nach außen gehen nur 80 und 10443 (Schritt 7).

System aktualisieren

Erst alles auf den neuesten Stand bringen.

auf dem Pi
sudo apt-get update && sudo apt-get upgrade -y
sudo apt-get install -y git curl
sudo systemctl enable --now systemd-timesyncd   # korrekte Uhrzeit (wichtig für TLS!)
⏰
Eine korrekte Systemzeit ist Pflicht — Let's-Encrypt-Zertifikate werden sonst als „nicht gültig" abgelehnt. Der NTP-Dienst oben erledigt das automatisch.

Feste IP-Adresse vergeben

Damit die FritzBox-Portfreigabe immer das richtige Gerät trifft.

Der Pi braucht eine gleichbleibende interne IP. Zwei Wege — wähle einen:

Variante A — empfohlen: feste Zuweisung in der FritzBox

In der FritzBox unter Heimnetz → Netzwerk → beim Gerät stechertennis auf Bearbeiten → Häkchen bei „Diesem Netzwerkgerät immer die gleiche IPv4-Adresse zuweisen". Fertig, kein Eingriff am Pi nötig.

Variante B — statisch direkt am Pi

auf dem Pi
sudo nmtui
# → Edit a connection → Wired → IPv4 CONFIGURATION = Manual
#   Addresses: 192.168.1.213/24
#   Gateway:   192.168.1.1   (IP deiner FritzBox)
#   DNS:       8.8.8.8
sudo systemctl restart NetworkManager
hostname -I   # prüfen
📝
Merke dir die IP (im Beispiel 192.168.1.213) — du brauchst sie gleich für die FritzBox-Portfreigabe.

Docker & Docker Compose installieren

Ein Befehl, danach einmal neu einloggen.

auf dem Pi
curl -fsSL https://get.docker.com | sh
sudo usermod -aG docker $(whoami)
exit                       # abmelden, damit die Gruppe greift

Neu einloggen und prüfen:

auf dem Pi (neue Sitzung)
ssh stecher@stechertennis.local
docker --version
docker compose version
sudo systemctl enable docker   # Autostart beim Booten
✅
Beide Versionsausgaben ohne Fehler = Docker läuft. docker run hello-world ist ein optionaler Zusatztest.
5

DuckDNS-Domain einrichten

Was wir von DuckDNS brauchen: Subdomain + Token.

  1. Auf duckdns.org einloggen (Google/GitHub/Reddit).
  2. Oben im Feld eine Subdomain eintragen, z. B. nechvatal → add domain. Deine öffentliche Adresse ist dann nechvatal.duckdns.org.
  3. Ganz oben steht dein token — eine 36-stellige UUID. Diese zwei Werte brauchst du:
WasBeispielWofür
Domain
(ohne .duckdns.org)		nechvatalApp-URL & Zertifikatsname
Tokenf5035eb4-6739-…-5ee827c7062dDNS-01-Challenge (Caddy holt Zertifikat) und IP-Updater
🔒
Der Token ist ein Geheimnis — er erlaubt das Ändern deiner DNS-Einträge. Nie in ein öffentliches Repo committen, nur in die .env (Schritt 9).
🌐
Warum DuckDNS? Deine Heim-IP von A1/Magenta wechselt. DuckDNS hält den Namen aktuell. Außerdem nutzt Caddy die DNS-01-Methode: das Zertifikat wird über einen DNS-Eintrag bestätigt — dafür ist der Token nötig, und Port 80 muss dafür nicht offen sein.
6

DuckDNS-Updater einrichten

Hält den Domain-Eintrag auf deine aktuelle Heim-IP aktuell.

Ein kleines Skript meldet alle 5 Minuten die aktuelle öffentliche IP an DuckDNS.

auf dem Pi
mkdir -p ~/duckdns && cd ~/duckdns
nano duck.sh

Inhalt (Domain & Token anpassen — ip= leer lassen, DuckDNS erkennt die IP selbst):

~/duckdns/duck.sh
echo url="https://www.duckdns.org/update?domains=nechvatal&token=DEIN-TOKEN&ip=&verbose=true" \
  | curl -k -o ~/duckdns/duck.log -K -
auf dem Pi
chmod 700 duck.sh
./duck.sh && cat duck.log     # muss "OK" zeigen

crontab -e
# diese Zeile ans Ende setzen — alle 5 Minuten aktualisieren:
*/5 * * * * /home/stecher/duckdns/duck.sh >/dev/null 2>&1

Test der Namensauflösung:

auf dem Pi
ping -c 3 nechvatal.duckdns.org   # sollte deine öffentliche IP zeigen
7

FritzBox: Portfreigaben

Damit das Internet den Caddy-Container erreicht.

Öffne http://fritz.box → Internet → Freigaben → Portfreigaben → Gerät für Freigaben hinzufügen und wähle den Pi (stechertennis). Lege zwei Regeln an:

BezeichnungProtokollPort externan Gerät (Pi)Port intern
HTTPS (App)TCP10443192.168.1.213443
HTTP→HTTPS RedirectTCP80192.168.1.21380
🔁
Warum 10443 statt 443 außen? Viele Provider/FritzBox-Setups belegen 443 selbst (Fernzugang). 10443 weicht dem aus. Deine öffentliche App-URL wird dadurch https://nechvatal.duckdns.org:10443. Innen lauscht Caddy weiterhin auf 443.
📶
Wichtig — keine DS-Lite-Leitung: Du brauchst eine öffentliche IPv4-Adresse. Prüfe in der FritzBox unter Internet → Online-Monitor. Steht dort nur eine IPv6-/DS-Lite-Adresse, ist die App von außen über IPv4 nicht erreichbar — dann beim Provider eine öffentliche IPv4 anfragen.
ℹ️
NAT-Hairpin: Der Aufruf der externen URL aus dem eigenen WLAN klappt bei manchen FritzBox-Modellen nicht (bekannte Router-Eigenheit). Teste dann über das Mobilfunknetz (WLAN am Handy aus).

App-Repository holen

Wir brauchen nur die Konfigurationsdateien — die Images kommen fertig von GHCR.

auf dem Pi
cd ~
git clone -b docker https://github.com/AINxtGenDev/stecher_tennis.git
cd stecher_tennis
📦
Die Container-Images (stecher-tennis-app und stecher-tennis-caddy) liegen multi-arch (amd64 + arm64) auf der GitHub Container Registry. Auf dem Pi wird automatisch die arm64-Variante gezogen — du musst nichts selbst bauen.

.env-Konfiguration ausfüllen

Das Herzstück — hier kommen DuckDNS-Domain, Token & Secret rein.

auf dem Pi
cp .env.example .env
python3 -c "import secrets; print(secrets.token_hex(32))"   # SECRET_KEY erzeugen
nano .env

Trage deine Werte ein. Beginne bewusst mit dem Staging-ACME (Testzertifikat), damit du bei Tippfehlern nicht ins Let's-Encrypt-Limit läufst:

.env
# --- Flask ---
SECRET_KEY=<hier den 64-Zeichen-Hex-Wert von oben>
CORS_ALLOWED_ORIGINS=https://nechvatal.duckdns.org:10443
FLASK_DEBUG=false

# --- HTTPS / DuckDNS ---
DUCKDNS_DOMAIN=nechvatal          # OHNE .duckdns.org
DUCKDNS_TOKEN=dein-36-stelliger-token
ACME_EMAIL=deine-email@example.com
HTTPS_PORT=10443

# Zuerst STAGING (Test), später auf production umstellen (Schritt 11):
ACME_CA=https://acme-staging-v02.api.letsencrypt.org/directory
VariableBedeutungMuss passen zu
SECRET_KEYVerschlüsselung der Sitzungen—
CORS_ALLOWED_ORIGINSerlaubte Browser-Herkunftexakt der User-URL inkl. :10443, kein Slash am Ende
DUCKDNS_DOMAINSubdomainSchritt 5
DUCKDNS_TOKENDNS-01-ChallengeSchritt 5
HTTPS_PORTexterner PortFritzBox-Regel (10443)
ACME_CAZertifikats-ServerStaging → später Production
⚠️
CORS_ALLOWED_ORIGINS muss zeichengenau mit der Adresse übereinstimmen, die im Browser steht — inklusive :10443. Sonst baut sich die Pyramide nicht auf (kein Live-Update). Das ist die häufigste Stolperfalle.

Container starten (mit Staging-Zertifikat)

Erster echter Start — Images ziehen und hochfahren.

im Ordner ~/stecher_tennis
docker compose pull          # lädt app- und caddy-Image (arm64)
docker compose up -d         # startet im Hintergrund
docker compose ps            # Status prüfen

Beide Container sollen laufen, die App als (healthy). Caddy holt jetzt im Hintergrund das Zertifikat über DuckDNS:

Logs ansehen
docker compose logs -f caddy   # Strg+C zum Beenden
# Erfolg: "certificate obtained successfully"
🧪
Ruf testweise https://nechvatal.duckdns.org:10443 auf. Beim Staging-Zertifikat warnt der Browser („nicht vertrauenswürdig") — das ist erwartet und beweist, dass die ganze Kette funktioniert. Klicke „trotzdem fortfahren" und prüfe, ob die App lädt.
11

Auf Produktions-Zertifikat umstellen

Erst wenn Staging einwandfrei lief — dann gibt's das „grüne Schloss".

im Ordner ~/stecher_tennis
# 1) ACME auf production umstellen
sed -i 's|acme-staging-v02|acme-v02|' .env

# 2) Stack stoppen
docker compose down

# 3) zwischengespeicherte Staging-Zertifikate löschen
docker volume rm $(docker volume ls -q | grep caddy_data)

# 4) mit echten Zertifikaten neu starten
docker compose up -d

Nach 1–2 Minuten ist https://nechvatal.duckdns.org:10443 ohne Warnung erreichbar — fertige Produktionsumgebung. 🎉

🔁
Nach jeder Änderung in .env gilt: docker compose down && docker compose up -d — ein bloßes restart liest die Datei nicht neu ein.

Funktionstest & erste Anmeldung

App läuft — jetzt Daten prüfen und absichern.

Die Datenbank wird beim ersten Start automatisch aus initial_players.json befüllt. Melde dich als Superadmin an und passe an:

  • Login öffnen, Standard-Zugangsdaten sofort ändern (Passwort-Reset im Superadmin-Bereich)
  • Unter /db_settings die Spielerliste prüfen / pflegen
  • Eine Test-Forderung anlegen und ein Ergebnis eintragen → Pyramide aktualisiert sich live
🔑
Ändere die Standard-Passwörter aus initial_players.json unbedingt vor dem öffentlichen Betrieb.

Backups, Auto-Updates & Wartung

Damit der Server unbeaufsichtigt zuverlässig läuft.

Datenbank-Backup per Cron

Die SQLite-DB liegt im Docker-Volume tennis_data. Ein einfaches, konsistentes Backup über den laufenden Container:

~/backup-db.sh
#!/bin/bash
cd ~/stecher_tennis
mkdir -p ~/db-backups
STAMP=$(date +%Y-%m-%dT%H-%M-%S)
# Python-Backup-API (im Image vorhanden) — konsistent auch im WAL-Modus.
# sqlite3-CLI gibt es im slim-Image NICHT, deshalb python -c:
docker compose exec -T app python -c "
import sqlite3
s=sqlite3.connect('/app/data/tennis.db'); d=sqlite3.connect('/app/data/backup-$STAMP.db')
s.backup(d); d.close(); s.close()"
docker compose cp app:/app/data/backup-$STAMP.db ~/db-backups/
docker compose exec -T app rm -f /app/data/backup-$STAMP.db
# nur die letzten 30 Backups behalten
ls -1t ~/db-backups/*.db | tail -n +31 | xargs -r rm
einrichten
chmod +x ~/backup-db.sh
crontab -e
# alle 3 Stunden sichern:
0 */3 * * * /home/stecher/backup-db.sh >/dev/null 2>&1

App aktualisieren (neue Version)

im Ordner ~/stecher_tennis
git pull                # falls sich compose/Caddyfile geändert hat
docker compose pull     # neues Image holen
docker compose up -d     # rollend neu starten

System-Updates & wöchentlicher Reboot

auf dem Pi
sudo apt-get install -y unattended-upgrades
sudo dpkg-reconfigure --priority=low unattended-upgrades

sudo crontab -e
# Sonntag 04:00 neu starten (Container starten dank restart-Policy automatisch):
0 4 * * 0 /usr/sbin/shutdown -r now
🛡️
Die Container starten dank restart: unless-stopped nach jedem Reboot von allein. Zusätzlich sind sie gehärtet: read-only Dateisystem, no-new-privileges und alle Linux-Capabilities entfernt (Caddy behält nur NET_BIND_SERVICE).

Fehlersuche

Die häufigsten Stolpersteine und ihre Lösung.

SymptomUrsache & Lösung
Seite nicht erreichbar (von außen) FritzBox-Portfreigabe prüfen (10443→443, 80→80); öffentliche IPv4 vorhanden? (kein DS-Lite); DuckDNS-IP aktuell? ping nechvatal.duckdns.org
TLS-/Zertifikatsfehler caddy_data-Volume löschen & neu starten:
docker compose down && docker volume rm $(docker volume ls -q | grep caddy_data) && docker compose up -d
Pyramide baut sich nicht auf CORS_ALLOWED_ORIGINS muss exakt der Browser-URL entsprechen (inkl. :10443, kein Slash). Danach down && up -d.
„too many certificates" von Let's Encrypt Du hast zu oft mit Production getestet. Mit acme-staging-v02 weiterarbeiten, ~1 Woche warten.
Container startet nicht / unhealthy docker compose logs app bzw. logs caddy ansehen; meist fehlender/fehlerhafter .env-Wert.
Eigene URL klappt nur vom Handy NAT-Hairpin der FritzBox — kein Fehler der App. Von außen (Mobilfunk) testen.
