◆ OVERVIEW

How GSD Fits Into Your Workflow

GSD structures your work into milestones (release cycles) containing phases (features/changes) broken down into plans (task lists). For smaller changes, you can skip the ceremony and use quick mode.

1

Decide the Scope

Is this a one-off tweak or a meaningful feature? Pick the right path: /gsd:quick for small tasks, /gsd:add-phase for bigger changes.

2

Discuss & Plan

Clarify implementation decisions, then GSD creates a structured task breakdown with verification criteria.

3

Execute

GSD agents implement each task with atomic commits. Every change is tracked and recoverable.

4

Verify & Ship

Automated verification checks that the goal was achieved, not just that tasks completed.

★ STEP 1

Decide: Quick Task or Phase?

Not every change needs the full ceremony. Use this decision matrix to pick the right approach for your stecher_tennis project.

Change Type Example Approach
Bug fix, typo, config tweak Fix score display rounding QUICK
Small feature (1-3 files) Add match duration tracking QUICK
Medium feature (4-10 files) Player statistics dashboard PHASE
Cross-cutting concern Add authentication layer PHASE
Multiple related features Season management + rankings MILESTONE
✓
Rule of thumb: If you can describe the change in one sentence and it touches fewer than 3 files, use /gsd:quick. Otherwise, use a phase.
⚡ STEP 2A

Path A: Quick Tasks

The fastest way to ship a small change. One command, minimal overhead, full GSD guarantees (atomic commits, state tracking).

Basic Quick Task

/gsd:quick "Fix the tiebreak scoring logic to handle 7-6 correctly"

GSD will plan and execute the fix in a single flow. No discussion, no research.

Quick Task with Extras

Add flags when you want more rigor:

# Want to discuss approach first? /gsd:quick --discuss "Add player ranking calculation" # Need GSD to research best practices first? /gsd:quick --research "Add WebSocket for live scores" # Want full verification after completion? /gsd:quick --full "Refactor match model validation" # Combine all three for maximum rigor /gsd:quick --discuss --research --full "Add match history export"
i
Quick tasks create their own directory in .planning/quick/ so they are tracked but don't clutter your main roadmap.
◉ STEP 2B

Path B: Set Up a Milestone

For bigger changes, you need structure. If GSD hasn't been initialized on your project yet, start here. Otherwise skip to adding phases.

First-Time Setup (One-Time)

If this is your first time using GSD on stecher_tennis:

# Navigate to your project cd ~/stecher_tennis # Initialize GSD — it will detect existing code /gsd:new-project
  • GSD scans your codebase and asks clarifying questions
  • Creates PROJECT.md capturing your vision and constraints
  • Defines requirements with traceable REQ-IDs
  • Generates a roadmap with phases and success criteria

Starting a New Feature Set

Already set up? Add a new milestone for your next batch of features:

# Start a new milestone (v1.1, v2.0, etc.) /gsd:new-milestone

This inherits your existing project context and extends the roadmap without re-asking questions you've already answered.

Adding a Single Phase

Just need to add one feature to the current milestone?

# Add a phase to the existing roadmap /gsd:add-phase "Add tournament bracket visualization"
⚙ STEP 3

Discuss: Clarify the Approach

Before GSD plans anything, it identifies "gray areas" — implementation decisions where your preference matters. This prevents rework.

Run Discussion

# For a specific phase number /gsd:discuss-phase 4 # Auto-mode: Claude picks sensible defaults for each question /gsd:discuss-phase 4 --auto

What It Asks

Layout choices, error handling strategy, API design patterns, data modeling decisions, UX flows

What It Produces

A CONTEXT.md file with locked decisions that downstream agents must follow

!
Skip discussion with --auto if you trust Claude's defaults. The discuss step is most valuable when you have specific opinions about how something should be built.
✎ STEP 4

Plan: Create the Task Breakdown

GSD generates detailed, executable plans with acceptance criteria for each task. Plans are verified before execution begins.

Generate Plans

# Plan a phase (reads CONTEXT.md from discussion) /gsd:plan-phase 4 # Include research step for unfamiliar territory /gsd:plan-phase 4 --research # After verification finds gaps, create fix plans /gsd:plan-phase 4 --gaps

What a Plan Contains

  • Wave assignment — which plans can run in parallel
  • File dependencies — files the executor must read first
  • Tasks with acceptance criteria (grep-verifiable conditions)
  • Must-haves — observable truths proving the goal was achieved
  • Requirement tracing — which REQ-IDs each plan addresses
✓
A plan-checker agent automatically reviews the plan quality before you execute. If it finds issues, GSD iterates up to 3 times to fix them.
▶ STEP 5

Execute: Build It

GSD agents implement each plan task-by-task with atomic commits. Parallel plans run simultaneously for speed.

Run Execution

# Execute all plans in a phase /gsd:execute-phase 4

This is where the actual code changes happen. Each task produces an atomic commit with a descriptive message.

Atomic Commits

Every task = one commit. If something breaks mid-phase, you can roll back to any task boundary.

Checkpoints

Some tasks pause for your confirmation (visual checks, architecture decisions, manual actions like OAuth setup).

Parallel Waves

Independent plans execute simultaneously. Dependent plans wait for their prerequisites.

Auto-Recovery

If a task fails verification, GSD retries with adjusted approach (up to 2 attempts) before escalating.

!
Context resets? If your session expires mid-execution, run /gsd:resume-work to pick up exactly where you left off.
✔ STEP 6

Verify: Confirm It Works

Verification is automatic after execution. GSD checks that the phase goal was achieved — not just that tasks were completed.

What Gets Verified

  • Observable truths — e.g., "User can view match history"
  • Artifacts exist — files created, imported, and wired together
  • Key connections — component calls API, API queries DB
  • Requirement coverage — each REQ-ID satisfied
  • Anti-patterns — scans for TODOs, stubs, empty handlers

If Gaps Are Found

# GSD identifies gaps and suggests fix plans /gsd:plan-phase 4 --gaps # Then re-execute to close the gaps /gsd:execute-phase 4

Gap closure creates targeted fix plans for specific issues — no need to re-plan the entire phase.

★ TIPS

Pro Tips & Useful Commands

Shortcuts and patterns for working efficiently with GSD on stecher_tennis.

Essential Commands Cheat Sheet

# Check progress and get routed to next action /gsd:progress # Natural language routing ("just tell GSD what you want") /gsd:do "add a feature to export match results as CSV" # Pause work with full context preservation /gsd:pause-work # Resume in a new session /gsd:resume-work # Run autonomous mode (discuss+plan+execute per phase) /gsd:autonomous # Insert an urgent fix between existing phases /gsd:insert-phase "Fix critical scoring bug" # Capture an idea without acting on it /gsd:note "Consider adding doubles match support" # Check and work on pending ideas /gsd:check-todos

Use /gsd:do

When in doubt, just describe what you want in plain English. GSD routes it to the right command automatically.

Use /gsd:progress

Start every session with this. It shows where you are and suggests the next action based on project state.

Use /gsd:pause-work

Always pause before ending a session. This creates a handoff file so the next session has full context.

Use /gsd:debug

For stubborn bugs, GSD's debug mode uses scientific method with persistent state across session resets.

/gsd:progress → /gsd:quick "your change" → done.
