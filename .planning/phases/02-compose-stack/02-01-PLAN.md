```yaml
services:
  app:
    build: .
    image: tennis-app:latest
    networks:
      - tennis_net
    volumes:
      - tennis_data:/app/data
    env_file: .env
    restart: unless-stopped
    # No ports: block — port 5000 only reachable inside Docker network (COMP-02)

  caddy:
    image: caddy:2-alpine
    networks:
      - tennis_net
    ports:
      - "80:80"
    volumes:
      - ./Caddyfile:/etc/caddy/Caddyfile:ro
      - caddy_data:/data
    restart: unless-stopped
    depends_on:
      app:
        condition: service_healthy

networks:
  tennis_net:
    driver: bridge

volumes:
  tennis_data:   # SQLite persistence — survives compose down (COMP-03)
  caddy_data:    # Caddy state — needed for Phase 3 TLS certs
```

Key points:
- No `version:` key — deprecated in Compose v2, omit entirely
- App service has `build: .` AND `image: tennis-app:latest` — build provides fallback if image not cached, image names the result
- App service has NO `ports:` block — satisfies COMP-02 (app only reachable via Caddy on internal network)
- Caddy publishes `80:80` — the only host-accessible port
- Both services on `tennis_net` custom bridge network — Docker DNS resolves `app` hostname for Caddy
- `tennis_data` named volume at `/app/data` — matches Dockerfile `DB_PATH=/app/data/tennis.db` (COMP-03)
- `caddy_data` named volume at `/data` — stores Caddy state, needed for Phase 3 TLS certificates
- `depends_on: app: condition: service_healthy` — Caddy waits for app's Dockerfile HEALTHCHECK to pass
- `restart: unless-stopped` on both — survives daemon restart, stays stopped after manual stop
- `env_file: .env` on app only — Caddy needs no env vars in Phase 2

  <action>
Create `Caddyfile` in the project root with this exact content:

```
# Phase 2: Plain HTTP reverse proxy — no TLS (Phase 3 adds DuckDNS ACME)
:80 {
    log {
        output stderr
        level ERROR
    }

    reverse_proxy app:5000 {
        header_up Host              {http.request.host}
        header_up X-Real-IP        {http.request.remote}
        header_up X-Forwarded-For  {http.request.remote}
        header_up X-Forwarded-Proto {http.request.scheme}
    }
}
```

Key points:
- `:80` site address (not `localhost:80` or a domain) — explicitly disables Caddy's auto-HTTPS, which would attempt TLS cert acquisition
- `log` block with `output stderr` and `level ERROR` — errors go to `docker logs` stderr stream, minimal noise
- `reverse_proxy app:5000` — `app` is the Docker Compose service name, resolved by Docker DNS on the shared `tennis_net` network
- Four `header_up` directives: Host, X-Real-IP, X-Forwarded-For, X-Forwarded-Proto — these are consumed by Flask's ProxyFix middleware (already configured in app.py with x_for=1, x_proto=1, x_host=1)
- Caddy 2 automatically handles WebSocket upgrade (Connection/Upgrade headers) — no explicit WS config needed
- This matches the existing RPi Caddyfile pattern from documentation/99_rpi.txt but targets `app:5000` (Docker DNS) instead of `localhost:8000`
